## 기본 원칙
- 작업 범위는 현재 프로젝트 루트 이하로만 한정한다.
- 프로젝트 내부 파일을 직접 읽고 판단한다.
- 추정으로 구현하지 않는다.
- 요구사항이 모호하면 먼저 짧게 확인하고, 확인이 어려우면 안전한 최소 범위로 작업한다.
- 필요한 경우 먼저 짧은 계획을 세운 뒤 작업한다.
- 코드 수정 후 관련 테스트를 우선 수행하고, 불가하면 최소 검증과 사유를 기록한다.
- 테스트를 실행할 수 없으면 그 이유를 명확히 기록한다.
- 변경 내용과 검증 결과는 마지막에 짧게 요약한다.
- `.env`, secret, credential, API key, 인증 파일은 출력하거나 수정하지 않는다.
- 프로젝트 스코프 밖 파일은 읽거나 수정하지 않는다.
- 삭제, 강제 덮어쓰기, 원격 push 같은 파괴적 작업은 수행하지 않는다.
- 사용자가 제공한 변수명, 함수명, 파일명은 임의로 바꾸지 않는다.
- 불필요한 대규모 리팩토링은 하지 않는다.
- 요청 범위 밖의 기능은 임의로 추가하지 않는다.
- 클래스에는 JavaDoc을 반드시 작성한다.
- 이후 Codex의 설명, 계획, 보고는 기본적으로 한국어로 작성한다.

## 형식 규칙
- 포맷터와 린터가 있으면 설정 파일과 실행 방법을 함께 유지한다.
- 규칙이 정해지지 않은 부분은 추정하지 말고, 기존 코드 스타일을 따른다.

## 테스트 원칙
- 관련 테스트는 기본적으로 반드시 추가하거나 갱신한다.
- 테스트를 못 돌리면 원인을 함께 적는다.

## 커밋과 PR
- 커밋 메시지는 짧고 명령형으로 쓴다. 예: `Add input validation`
- PR에는 변경 요약, 검증 방법, 관련 이슈를 적는다.
- UI 변경이 있으면 스크린샷을 함께 둔다.

## 작업 메모
- 변경은 작고 명확하게 유지한다.
- 새 규칙이 필요하면 이 문서를 먼저 갱신한 뒤 작업한다.
